# 대량 발행 시스템의 운영 안정성을 개선하는 백엔드 개발자

> 수천 개 고객사가 사용하는 전자세금계산서 서비스에서  
> 월 120만 건 규모의 발행 흐름, 에이전트 연동, 보안 패치, AI 기반 운영 자동화를 경험했습니다.

`Java` `Spring Framework` `JSP` `MyBatis` `Oracle` `MySQL` `REST API` `Linux` `Apache/Tomcat` `Git` `AI Integration`

---

## 핵심 임팩트

### 수천 개 고객사 운영 환경 경험

전자세금계산서 서비스의 발행, 역발행, 국세청 전송, 매입/매출 보관함, 휴폐업 조회 흐름을 운영했습니다.  
월 120만 건 규모의 발행 데이터와 월초 집중 기간의 대량 발행 흐름을 다루며, 장애 발생 시 코드·DB·서버 설정·고객사 환경을 함께 확인했습니다.

### 발행 집중 기간 대응

계산서 발행이 몰리는 기간에는 일주일 기준 50만 건 이상, 월초 1일~10일 기준 최대 100만 건 수준까지 발행량이 증가했습니다.  
이 시기에는 발행 누락, 에이전트 동기화 실패, 인증서 오류, 고객사 서버 환경 이슈를 우선순위에 따라 분석하고 운영팀과 공유했습니다.

### 발행현황 조회 성능 개선

발행현황 화면의 조회 응답 시간을 약 10초에서 3초 수준으로 줄였습니다.  
Oracle Materialized View를 활용해 반복 조회되는 운영성 데이터를 빠르게 확인할 수 있도록 개선했고, 운영자가 매출과 발행량을 한눈에 파악할 수 있게 했습니다.

### 운영 배포와 보고 업무 자동화

파일질라로 클래스 파일을 직접 업로드하던 구식 운영 반영 절차를 AI 기반 보조 시스템으로 일부 자동화했습니다.  
배포 대상 확인, 반영 전 점검, 반복 정리 작업을 줄여 운영 반영 시간을 비약적으로 단축했고, 일일 보고 데이터를 엑셀로 내보내는 프로그램을 만들어 반복 보고 업무도 줄였습니다. CVS 기반 형상관리도 Git으로 전환해 변경 이력과 운영 반영 흐름을 더 추적 가능하게 만들었습니다.

## 제가 해결해온 문제

### 1. 대량 발행 데이터와 정합성

**문제**  
세금계산서 발행 누락, 대량발행 오류, 재동기화, 국세청 전송 상태 확인처럼 운영 데이터의 정합성이 중요한 이슈가 반복되었습니다.

**접근**  
발행 요청, DB 상태, 문서 상태, 국세청 전송 흐름을 기준으로 어디서 실패했는지 추적했습니다. 단순히 화면 오류만 보는 것이 아니라 데이터가 생성되고 전송되는 전체 흐름을 기준으로 원인을 좁혔습니다.

**기술**  
`Java` `MyBatis` `Oracle` `MySQL` `SQL` `Data Sync`

**강점**  
CRUD 기능 구현보다 운영 데이터의 상태 변화, 재처리 기준, 장애 복구 흐름을 이해하는 데 강점이 있습니다.

### 2. 고객사 에이전트 연동 장애 분석

**문제**  
월 10건 내외로 고객사 서버의 에이전트 설치, 이전, 오류 대응이 발생했습니다. 주요 이슈는 에이전트 동기화 실패였고, 원인은 고객사 네트워크, DB 연결, JDK 버전, 드라이버, 설정값 등으로 다양했습니다.

**접근**  
고객사 서버에 설치된 에이전트 환경을 확인하고, 고객사 DB 스펙과 연결 조건을 분석해 연동 가능 여부를 판단했습니다. 로그, 설정 파일, DB 접속 조건, 실행 환경을 함께 확인했습니다.

**기술**  
`Linux` `Agent` `JDBC Driver` `JDK` `DB Schema` `Log Analysis`

**강점**  
백엔드 코드 안에서만 문제를 보지 않고, 실제 운영 환경과 고객사 인프라 차이까지 고려해 장애 원인을 찾을 수 있습니다.

### 3. AI 기반 운영 자동화와 배포 체계 개선

**문제**  
운영자가 매주 발행량과 매출을 수동 집계해야 했고, 운영 반영도 파일질라로 클래스 파일을 직접 올리는 방식이었습니다. 배포 대상 확인, 수동 업로드, 결과 정리까지 사람이 반복 처리해야 해서 시간이 오래 걸리고 실수 위험이 컸습니다.

**접근**  
관리자 화면에서 매출과 발행량을 확인할 수 있도록 개선하고, 스케줄러 기반 SMS 자동 발송으로 주요 지표를 정기적으로 공유하도록 구성했습니다.  
또한 AI를 활용한 운영 반영 보조 시스템을 만들어 배포 준비와 확인 과정을 일부 자동화했습니다. 일일 보고 데이터를 엑셀로 내보내는 프로그램을 개발했고, CVS로 관리되던 프로젝트를 Git 기반으로 전환해 변경 이력 추적과 협업 흐름을 개선했습니다.

**기술**  
`Java` `Admin` `Scheduler` `SMS` `Excel Export` `AI Workflow` `Git Migration` `Oracle MView` `SQL Tuning`

**강점**  
반복적이고 비효율적인 운영 업무를 그대로 감수하지 않고 자동화 대상으로 분해합니다. 레거시 배포, 보고, 형상관리처럼 팀의 속도를 늦추는 업무를 찾아 실제 운영 시간이 줄어드는 도구로 바꾸는 데 강점이 있습니다.

### 4. 보안 패치와 라이브 서버 반영

**문제**  
SSL/TLS, SQL Injection, Apache 버전 취약점처럼 운영 서비스에 직접 영향을 줄 수 있는 보안 조치가 필요했습니다.

**접근**  
영향도 분석, 설정 변경, 테스트, 배포, 문서화까지 진행했습니다. 특히 Apache 2.4.6에서 2.4.64로 업그레이드한 뒤 라이브 서버에 반영했고, 기존 mod_jk 기반 WAS 연동 구조의 호환성을 함께 확인했습니다.

**기술**  
`Apache` `Tomcat` `mod_jk` `SSL/TLS` `DNS DCV` `SQL Injection 차단`

**강점**  
보안 점검 결과를 실제 운영 작업으로 쪼개고, 서비스 영향도를 확인하면서 반영할 수 있습니다.

### 5. AI 기능 연동 개발

**문제**  
AI 기반 세금계산서 발행 기능을 기존 발행 시스템 안에서 사용할 수 있도록 프론트 화면과 Java 백엔드 연동이 필요했습니다.

**접근**  
LLM 자체를 구축한 것은 아니지만, AI 처리 결과를 사용자가 확인하고 발행 흐름으로 이어갈 수 있도록 화면과 백엔드 연결부를 담당했습니다. 기존 세금계산서 발행 흐름과 충돌하지 않도록 입력/출력 데이터, 예외 상황, 운영 테스트 흐름을 확인했습니다.

**기술**  
`Java` `JSP/Frontend` `REST API` `AI Integration` `QA Checklist`

**강점**  
새로운 AI 기능을 과장 없이 운영 서비스에 붙이는 관점에서 접근할 수 있습니다. 모델 자체보다 중요한 입력/출력 검증, 기존 업무 흐름과의 연결, 운영팀이 사용할 수 있는 상태로 다듬는 역할을 수행했습니다.

### 6. CVS에서 Git으로 전환

**문제**  
기존 형상관리가 CVS 기반으로 운영되어 변경 이력 추적, 협업, 운영 반영 전후 비교가 불편했습니다.

**접근**  
기존 프로젝트를 Git 기반으로 사용할 수 있도록 전환하고, 운영 반영과 변경 관리 흐름을 더 추적 가능한 방향으로 개선했습니다.

**기술**  
`CVS` `Git` `Release Note` `Documentation`

**강점**  
오래된 개발·운영 방식을 그대로 두지 않고, 팀이 더 안전하게 변경 사항을 관리할 수 있는 방향으로 개선합니다.
